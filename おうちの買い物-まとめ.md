# おうちの買い物 ・ 企画から公開までのまとめ

家族で共有する買い物・在庫管理アプリの、検討内容・決定事項・成果物・公開手順を1つにまとめたものです。

---

## 1. 何を作るか

家族全員が各自のスマホから使える、**買い物リスト＋在庫（常備品）管理**のWebアプリのPWA。
誰かがリストに追加すると、店にいる家族のスマホへリアルタイムに反映されるのが中心の価値。

- **対象端末**：家族のスマホはAndroidのみ。共有iPadあり（iOSは動作確認程度で、重要度は低い）
- **最優先の方針**：とにかくストレスなく使えること
- **土台**：以前作った野菜家庭菜園アプリと同じ「単一HTML＋Netlify系デプロイ」の流儀を踏襲

---

## 2. 検討で決めたこと（と、その理由）

| 論点 | 決定 | 理由 |
|---|---|---|
| 家族での共有方式 | **各自のスマホで共有（要同期）** | 「頼まれた物が店で見える」が家族アプリの本命 |
| 同期の仕組み | **Firebase（Firestore）** | リアルタイム同期が標準。単一HTMLのまま使える。無料枠で足りる |
| 通知の鳴らし方 | **切り替え式トグル（初期＝お使い依頼型）** | 「多い／少ない」は使ってみないと分からない。机上で1つに固定せず、後で調整できる形が正解 |
| 通知の配信 | **今は未接続（Phase 2で有効化）** | まずリスト同期を動かし、家族の体感を掴んでから配信を組む |
| レシート写真での更新 | **保留（Phase 2）** | 「確認前提・補充ループ限定」なら実現可能だが、全自動は事故率が上がる。核が固まってから |
| 配布形態 | **インストール型PWA** | Androidは通知・オフラインが自然。ホーム画面から起動できる |
| 設定の入力 | **コード編集不要（起動画面で入力）** | 個人PCにエディタが無くても運用できるように |

### 通知スタイルの3案（設定でいつでも切替）
- **お使い依頼型（初期値）**：「お使いを頼む」を押した時だけ、今のリストを1回まとめて送る。いちばん静か
- **まとめ型**：追加を少しためて、間引いて自動でお知らせ
- **「切れた」だけ**：常備品が切れた時のみ通知。それ以外はアプリ内バッジで気づかせる

## 3. 機能一覧（Phase 1・実装済み）

- **買い物リスト**
  - 売り場順のカテゴリ表示（野菜→肉魚→乳製品→…→日用品）で、店内を上から回れる
  - **店フィルタ [すべて／スーパー／ドラッグ]**（v1.7.0〜）。食料品＝スーパー、日用品＝ドラッグ、調味料・その他＝中間（両方に表示）。選択は記憶される
  - **売り場の並び順を変更できる**（v1.8.0〜）。設定の「売り場の並び順」で▲▼でカテゴリ順を入れ替え＝お店のレイアウトに合わせられる。買い物リストと在庫の並びに反映、記憶される
  - **買ったものを「まとめて消す」**（v1.9.0〜）。チェック済みを1タップで一括消去（確認あり）
  - **「あとで買う」置き場**（v1.10.0〜）。「無くなったけど今すぐ買わない」品は各行の「あとで」で💤グループへ（薄表示・件数外）。「買う」で復帰。再追加や「切れた」でも自動復帰
  - 1品目1行のコンパクト表示（名前と「誰が・いつ」を横並び）で、1画面に多く表示（v1.2.0〜）
  - ページ下部に「定番・常備品から追加」候補を**全件**折り返し表示。肉・野菜を先頭、油など調味料を最後に並べ替え。リストに入れた物は候補から消える（v1.2.0〜）
  - 入力した品名からカテゴリを自動推定。追加バーの絵文字チップをタップすれば手動でジャンルを選び直せる（v1.1.0〜）
  - **各行の左の絵文字バッジをタップすると、その品のジャンルを後から変更できる**（v1.5.0〜）。変更はその品だけに反映
  - チェックで「買った」に。誰がいつ追加・購入したかを表示
- **在庫（常備品）**
  - 3状態のセグメント：**ある／少ない／切れた**（数値入力なし）
  - 「切れた」→ 買い物リストへ自動追加
  - リストで「買った」→ 同名の常備品が自動で「ある」に戻る（閉ループ）
  - 各行のゴミ箱ボタンで常備品を削除（確認あり）。v1.2.1〜。以前は0.6秒長押しの隠し操作で、削除できないと分かりにくかったため改善
  - 追加時にジャンルを手動選択可能（v1.1.0〜、買い物リストと同じ絵文字チップ方式）
  - 各行の絵文字バッジをタップしてジャンルを後から変更できる（v1.5.0〜。例：にんにくを野菜→調味料へ）
  - **初回だけ、常備品リストに定番一式（約80品）を自動登録**（v1.3.0〜）。手入力の手間を省く。`STOCK_SEED`（明示カテゴリ付き）が定義元で、`COMMON`（サジェスト候補）もこれと同じ品揃え。判定は「その家族の常備品がサーバー上で空のときだけ」＋端末の`kaimono.seeded.<家族コード>`フラグで二重登録を防止
- **更新の履歴**：誰が・何を・いつ、を時系列で表示
- **お使いを頼む**：今のリストを家族に知らせる（配信はPhase 2）
- **バックアップ**：全データをJSONで書き出し／読み込み。さらに**「バックアップを共有」**（v1.6.0〜）で、Androidの共有シートからGoogleドライブ・メール等へ1タップ退避できる（共有非対応の環境では自動でダウンロードにフォールバック）。消えても「読み込む」で復元
- **PWA**：ホーム画面に追加、オフラインでも起動

## 4. 技術構成

- **単一HTMLアプリ**（`index.html`）＋ PWA用の少数ファイル
- **Firebase Firestore**：リアルタイム同期＋オフライン保存（IndexedDBキャッシュ）
- **Firebase Authentication（匿名ログイン）**：セキュリティルールの前提として使用
- **データ構造**：`families/{家族コード}/list`, `/stock`, `/history`
- **本人情報**（名前・色）と**Firebase設定**はスマホ側のlocalStorageに保存（コードには書かない）
- **Service Worker**：アイコン等の外枠のみキャッシュ優先。**index.html（アプリ本体）はネットワーク優先**にし、オフライン時だけキャッシュへフォールバック（v1.1.3〜。以前はHTMLもキャッシュ優先だったため、更新してもスマホに反映されない不具合があった）

## 6. 公開手順

### 6-1. Firebase の準備（共通）
1. Firebaseコンソールで無料プロジェクトを作成
2. **Firestore Database** を作成（リージョンは asia-northeast1 など）
3. **Authentication** で **匿名ログイン** を有効化
4. Firestoreの「ルール」に `firestore.rules` を貼って公開
5. プロジェクト設定でWebアプリを追加 → `firebaseConfig` をコピー
6. **Authentication ＞ 承認済みドメイン** に、公開先のドメインを追加（※忘れると繋がらない）

### 6-2. アプリを開いて初期設定
公開URLを開く → 起動画面で `firebaseConfig` を貼り、名前・色・**家族コード**を入力 → 「はじめる」。
家族には**同じ公開URLと同じ家族コード**を伝えれば、同じリストに繋がる。

## 7. ホスティングの選択肢

いずれも無料・HTTPS対応。アプリは相対パスで作ってあるので、どれでも修正不要。

| | Netlifyドロップ | GitHub Pages | GitHub＋Netlify連携 |
|---|---|---|---|
| 公開方法 | フォルダをドラッグ＆ドロップ | リポジトリ作成→配置→Settingsで有効化 | pushで自動デプロイ |
| 更新 | 再ドロップ | `git push` | `git push` |
| 履歴管理 | なし | Gitで残る | Gitで残る |
| URL | `xxx.netlify.app` | `ユーザー名.github.io/リポジトリ名/` | ルートURL |
| 無料の条件 | 特になし | リポジトリを**公開**にする必要あり（privateはPro/有料） | 同左 |

- **「リポジトリが公開になる」問題は無い**：機密になりうるFirebase設定はコードに書かず起動画面で入力する方式。仮に書いても、Firebaseのウェブ設定は秘密情報ではなく、守りはセキュリティルール側で行う設計
- **サブフォルダURLでもPWAは動く**：相対パスで組んであるため
- **おすすめ**：普段からGit/GitHubを使うなら、バージョン履歴が残るGitHub Pages（または連携）が向いている

## 8. 今後の実装ロードマップ（2026-07-05 に方針決定）

家族から出た「入れたい機能」を整理したもの。**実装順は上から**。①②③のA案は既存を壊さず軽く積める。重い部分（②B・③B）は後回し。

### ① 店で分ける（スーパー / ドラッグストア）　難易度：普通　【✅ 完了 v1.7.0】
- 前提として、**既存の品のジャンルを後から直せる機能を先に実装**（v1.5.0。各行の絵文字バッジ→9カテゴリ選択）。にんにく＝野菜→調味料 のような修正ができる＝店の振り分けの精度を保てる
- **実装済み（v1.7.0）**：買い物リスト上部に [すべて／スーパー／ドラッグ] の店フィルタ。`STORE_OF`（カテゴリ→店）で振り分け。食料品=スーパー、日用品=ドラッグ、調味料・その他=中間（どちらの店でも表示）。選択は記憶。中間の見せ方はA案（両方に表示）を採用
- 食料品（野菜・肉・乳製品・パン・冷凍・お菓子）＝**スーパー**、日用品（洗剤・ペーパー類）＝**ドラッグストア**、調味料（減りが遅い）＝**中間（どちらでも）**
- 実装：今の9カテゴリの上に「店」の対応表を1枚かぶせる。買い物リスト上部に切り替え **[すべて][スーパー][ドラッグストア]** を置き、選んだ店の品だけ表示。調味料は中間として両方に出す
- データ構造は変えず、対応表＋絞り込み表示を足すだけ。②の土台にもなる

### ② 並び替え（カードのように）　【まずA、後でB】
- **A案：カテゴリ（売り場）の順番を並び替え**　難易度：普通　【✅ 完了 v1.8.0】
  - 近所の店のレイアウトに合わせてカテゴリ順を自分で決められる。設定の「売り場の並び順」で▲▼で入れ替え、`catOrder`をsettingsに保存、買い物リスト・在庫に反映。このアプリの「上から回れる」価値を各家庭に最適化
  - 実装は▲▼ボタン方式（タッチで確実）。B案の自由ドラッグは将来課題のまま
- **B案（将来）：品目1つずつを自由にドラッグ**　難易度：難しい
  - 並び順の番号をデータに持たせる／スマホのタッチドラッグ（誤操作・実装が厄介）／複数人編集時の順番ズレ対応が必要。専用ライブラリ検討。今のカテゴリ自動整列とも競合するため後回し

### ③ 今ある食材からメニュー提案　【まずリンク、AIはPhase 2】
- **A案（先にやる）：レシピサイトへ外部リンク**　難易度：とても易しい
  - 在庫や選んだ食材をキーワードにクックパッド等の検索を開く（例 `https://cookpad.com/search/鶏肉 キャベツ`）。サーバー不要・費用ゼロ・低リスク
  - **要検討（家族の指摘 2026-07-05）**：在庫の品名が「肉」「葉物野菜」のように大まか＝レシピ検索に使いにくい。実装時は、レシピ用に**具体名（豚こま・鶏むね・ほうれん草 等）を選ばせる／持たせる**工夫が要る。候補：①検索前に品を具体化する簡易入力、②在庫に「レシピ用の具体名」を任意で持たせる、③選択した数品だけを対象にする。まずは「選んだ数品でクックパッド検索」から始め、粒度は運用しながら調整
- **B案（Phase 2）：AI（Gemini等）で献立提案**　難易度：やや難しい
  - 在庫食材をAIに渡し使い切り重視の献立を提案。APIキー保護に薄いサーバー（Netlify Functions等）が必要。**下のレシート読み取りと同じサーバー基盤**なのでPhase 2でまとめてやると効率的

### 補足：一度きりの買い物（ボールペン等）について
- **現状のままでOK・改修不要**。買い物リストの入力欄に品名を打てば、在庫（常備品）に登録せずリストに追加できる。買ってチェックすれば消える。一度きりの品は在庫を汚さない
- アプリの本質（＝「買うものリスト」）からズレない。カテゴリは自動推定＝「その他」等になるが、絵文字バッジ（v1.5.0）でいつでも変更可能
- ①の店分けでは「その他/日用品」はドラッグストア寄りに割り当て予定なので、ボールペン等もその流れで扱える

## 8-1. コードレビュー指摘と改良方針（2026-07-07）

### レビューで見つかった課題（未対応のもの）
- **履歴の書き込みが無限に溜まる**：読み込みはlimit(80)で対処済み（v1.9.0）だが、書き込みは削除されず蓄積。古い履歴の自動削除か間引きが今後必要
- **通知UIが「動くフリ」をしている**：「お使いを頼む」は配信未実装なのに「伝えました」と出る。Phase 2の配信実装まで「準備中」表示にすべき
- **初回シードの競合**：家族2人が同時に初回アクセスすると在庫が二重登録される理論上の穴（実運用ではほぼ起きない）
- バックアップに履歴・設定が含まれない／復元はFirestoreバッチ500件上限
- 売り場並び順・店フィルタは端末ごと保存（家族間で共有されない。現状は仕様として許容）
- 単一HTML 1,300行超の保守性／自動テストなし／confirm()ダイアログが無骨

### 改良方針（コンセプト：めんどくさがり夫婦がストレスなく）
1. **入力を減らす**：共有ターゲット化（他アプリの共有→リスト追加）、PWAショートカット、レシート読み取り（Phase 2）
2. **通知を本当に動かす**：OneSignal最有力（Phase 2の本丸）。それまで通知UIは「準備中」表示
3. **ストレス除去**：消去の「元に戻す」トースト、買ったものの自動掃除、アプリアイコンにバッジ（Badging API）
4. **資産保全**：履歴の自動掃除、バックアップへの履歴含め、月1バックアップリマインド

## 8-3. プッシュ通知・要件定義（2026-07-07 確定）

### 目的
「お使いを頼む」「切れた」が、アプリを閉じていても家族のスマホへ実際に届くようにする（Webプッシュ）。

### 決定済みの要件
| 項目 | 決定 |
|---|---|
| 対象端末 | **Androidスマホのみ**（PWAで受信。iPadは動けばラッキー・保証しない） |
| 通知モード | 既存の3モードをそのまま実装：①お使い依頼型（初期値）②まとめ型（5分バッファで間引き）③「切れた」だけ |
| モードの扱い | **受け取る人ごとの設定**（各自のスマホで受信量を選ぶ。例：夫は依頼だけ、妻は切れたも受け取る） |
| 通知の文面 | **品名を載せる**（例「買い物をお願い：牛乳、卵 ほか3件」。ロック画面表示OK、買い物なので機密性低） |
| 自分の操作 | **自分には通知しない**（頼んだ本人・切らした本人には送らない） |
| タップ時 | アプリ（PWA）が開く |
| 受信の準備 | 各自のスマホで1回だけ「通知を許可」する（設定タブに許可ボタンを置く） |
| 費用 | 無料枠内に収める |
| 失敗時 | 通知が届かなくてもアプリ本体の動作（同期）には影響させない |

### 保留（次に決める）：裏側の基盤
静的サイトに秘密鍵は置けないため、送信には裏側の仕掛けが必須。以下2案を比較検討中：

| | 案A：Firebase Cloud Functions＋FCM | 案B：OneSignal＋Netlify Functions |
|---|---|---|
| 仕組み | Firestore書き込みをトリガーに自動送信（Firebase内で完結） | 通知SaaS＋鍵を隠す中継サーバー |
| 新アカウント | 不要 | OneSignal＋Netlifyの2つ必要 |
| クレジットカード | **必要**（Blazeプラン。家族利用なら実質0円、予算アラートで暴発防止可） | 不要 |
| 構造 | ◎ 素直。後のレシートAI・献立AIにも流用可 | △ 中継の入口を家族コードで守る（防御薄め・買い物リストなので許容範囲） |
| 推し | カード登録に抵抗がなければこちら | カードを避けたいならこちら |

### 実装時の注意（要件から導かれる設計メモ）
- 受信者ごとのモード設定＝送信側でのフィルタが必要（FCMならトークンごとにFirestoreへ設定を保存／OneSignalならタグ）
- 「自分に送らない」＝送信者の識別子（uid）を除外条件に入れる
- まとめ型の5分バッファは送信側（サーバー側）で間引くのが確実（今のクライアント側タイマーは端末が閉じると動かない）
- 通知許可ボタンと許可状態の表示を設定タブに追加。未許可・拒否時の案内文言も用意
- 実装完了までは、現在の「お使いを頼む」の完了トースト（動くフリ）を「準備中」表示に直すこと

## 9. 設計上の注意点

- **家族コードは「知っていれば見られる」方式**。買い物リストなので機密性は低いが、推測されにくい言葉にして家族以外に共有しない
- 静的サイトなので、ホスティングを問わずコードは閲覧可能。だからこそ**秘密情報をコードに置かない設計**にしている
- より厳密にしたい場合は、家族ごとにメンバーを登録して所属チェックする方式へ拡張できる
- iOSはPWAプッシュが弱いが、iPadは動作確認用途のため、通知なしでも他機能が動けば十分

## 12. 開発用デモ（テスト）モード（v1.4.0〜）

- 公開URLやローカルに `?demo=1` を付けて開くと、**Firebaseを使わず端末内の仮データ**で動作する（例：`.../kaimono-app/?demo=1`）
- 接続情報（firebaseConfig）が無くてもUI・データの動き（初回登録・追加・削除・「切れた」→買い物リスト自動追加・買ったら常備品が「ある」に戻る 等）を確認できる。Claudeの動作確認や、実機で触って試すのに使う
- ヘッダーの同期表示は「デモ」になる。データは保存されず、リロードで初期状態（定番一式が入った状態）に戻る
- 通常利用（`?demo` なし）には一切影響しない。実装は各操作関数の先頭に `if(DEMO){…}` の早期分岐を置く形（本番のFirestore経路は無改変）

## 14. 定番リスト.csv の運用（v1.13.0〜、v1.14.0で小分類列を追加）

- リポジトリの `定番リスト.csv` が定番品のマスター。列：**品名／カテゴリ（日本語名）／常備（A=常備品・B=在庫のみ）／ざっくり（Z=汎用名）／小分類**
- UTF-8（BOM付き）なのでExcelでそのまま開ける
- 運用：Excelで編集して保存 →「反映して」とClaudeに伝えると、`index.html` の `STOCK_SEED`・`GENERIC`・`SUB_OVERRIDE` に同期してバージョンを上げてpushする
- アプリが実行時にCSVを読むことはしない（単一HTML方式の維持。読み込み失敗の考慮を不要にするため）
